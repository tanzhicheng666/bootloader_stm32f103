# bootinfo分区写入成功但是分区切换失败

## 😉Overview

### 1️⃣APP侧未正确完整修改bootinfo

1️⃣APP侧OTA状态机里面写`bootinfo`分区的时候没有修改**state字段**，导致虽然修改了`pending_app`，保存到flash了，但是`bootloader`侧的`根据状态选择运行分区的状态机完全失效`，state一直是**normal**,即使**pending_app修改了，但是active_app还一直是APP_A**，于是重启以后还是运行了`APP_A`.

2️⃣把`state`修改以后还不行，因为还要对新的固件进行CRC32的校验才能跳转，但是APP侧没有做APP_B_CRC字段的修改，这个地方也会出错。所以要在接口里面添加一个**固件CRC32的参数进来写进bootinfo**（固件CRC通过OTA模块的上下文传入）

```c
//提供给APP侧用于标记新固件Pending（第一次写忘记加CRC字段了）
void Bootloader_SetPending(uint32_t pending_app,uint32_t crc)
{
    boot_info_t bt;
    BootInfo_Load(&bt);//先获取当前bootinfo所有信息字段
    bt.pending_app=pending_app;//挂起新的分区
    //bt.state=OTA_STATE_PENDING;//🌟缺少这一行，修改bootinfo的state
    //bt.app_b_crc=crc;//🌟缺少CRC校验的修改
    BootInfo_Save(&bt);//保存写道对应的flash

}
```

### 2️⃣APP侧写入新固件的对齐

**flash写入设计的是逐字写入，应该考虑不能被4整除的情况（即使把payload固定成512也要考虑，因为最后一包的长度是不确定的，可能不到512，也可能不能被4整除）**

***🚨APP_A侧OTA模块状态机处理办法：通过（len+3）/4向上取整先计算出word数，借助临时缓冲区，大小设置成PAYLOAD_LEN+3，防止len=PAYLOAD_MAX_LEN的时候aligned缓冲区越界非法访问。***

把`ctx`的`payload`字段`memcpy`到`aligned_buf`缓冲区（这里在函数体内部声明这么大的数组容易导致栈溢出`hardfault`,所以应该加个`static`关键字）

```c
uint32_t word_count=(ctx.parser.len+3)/4;//向上取整
//使用临时缓冲区存放写入flash的数据：对齐，防止访问payload越界
static uint8_t aligned_buf[PAYLOAD_MAX_LEN + 3];
memset(aligned_buf,0,sizeof(aligned_buf));
memcpy(aligned_buf,ctx.parser.payload,ctx.parser.len);
//写入falsh
flash_write(START_ADDRESS+ctx.seq*PAYLOAD_MAX_LEN,(uint32_t*)aligned_buf,word_count);
```

**写flash需要四字节对齐，但是临时数组不需要，灵活性比较大，所以可以先转成对齐的临时数组，再写入falsh**

先对临时数组全部清零初始化`memset`，然后在把`ctx.patload`给`memcpy`进来（len界定），剩下的位置都是0，写入flash的时候就读取`aligned_buf`，传入之前算好的向上取整的`word_count`

***不能被4整除用0对齐***

**🌟不在上位机侧进行固件对齐处理，容易污染语义，写入flash对数据流有要求就在写之前处理**

### 3️⃣bootinfo侧CRC32新固件校验失败

**bootloader工程中原始的CRC校验函数**：校验的是整个分区，而非只有新固件的flash。

***但是，flash的默认存储的是0xFF，而不是0x00,这样进行CRC校验肯定结果不一样***

***此外还应该注意固件写入的对齐粒度应该怎么处理***

最开始的CRC32校验里面设计的是校验整个分区，这样不能精确的只对新固件的内容校验，而且也增加了计算量，比如说我使用的固件也才8k,但是完整的`app分区`有`96k`，所以在进行CRC校验的时候要传入固件的大小。

**还要注意**：**STM32的硬件CRC**传入的参数（或者指针）都是`32位`的，所以要计算`word_len`，这里记得结合之前对`flash分区`做的**补零对齐**。这里计算字数也是同样的做法，对`firmware_len`向上取整

代码示例：

```c
    uint32_t word_len=(firmware_size+3)/4;//进行向上取整的操作
```

### 4️⃣写flash对齐对CRC校验的影响怎么处理？

现在遇到了一个问题，写flash对固件进行了4字节对齐，bootloader在使用硬件CRC计算固件CRC的时候是以word为单位的，对齐操作会影响计算结果，导致和bootinfo里面的结果不一样，这样即使新固件没有问题，CRC校验也一直不能成功

***🌟为了语义更清晰，上位机不能再固件最后一包补零对齐。但是可以在计算CRC的时候动手脚：计算补零后的CRC32值，传递原固件内容保持不变不修改，固件写flash的对齐交给APP侧的OTA升级状态机***

硬件CRC校验接口

```c
calculated_crc = HAL_CRC_Calculate(&hcrc, (uint32_t*)appAddr, firmware_len_words);
```

### 5️⃣野指针非法访问（第二次出错了）

**错误代码示例：**声明一个指针变量，***没有初始化指向就使用***

```c
    //固件成功运行，标记有效
   boot_info_t* bt;        // 只声明了一个指针变量，没有让它指向任何有效地址，值是栈上的垂圾值（野指针）
    BootInfo_Load(bt);      // 把这个野指针传进去，BootInfo_Load内部大概率是往*bt里写数据（读flash后memcpy或逐字段赋值）
    if(bt->state==OTA_STATE_TESTING)  // 对野指针解引用，读到的也是垂圾内存
    {
    Bootloader_MarkValid();
    }
```

结果就是`野指针非法访问`导致`hardfault`难怪`新固件`看上去没有运行，没有反应，`但是新固件也没有运行`。

**✅正确使用示例：**

```c
    //固件成功运行，标记有效
    boot_info_t bt;
    BootInfo_Load(&bt);
    if(bt.state==OTA_STATE_TESTING)
    {
    Bootloader_MarkValid();
    }
```

**🛠️新固件标记有效：在关键代码执行以后标记有效，这里采用的是修改bootinfo结构体的state字段，从TESTING进入NORMAL同时清零boot_count,也可以单独加一个标志位判断**

```c
//新固件标记有效，从testing测试状态进入normal
void Bootloader_MarkValid(void)
{
    boot_info_t bt;
    BootInfo_Load(&bt);
    bt.state=OTA_STATE_NORMAL;
    bt.boot_count=0;
    BootInfo_Save(&bt);
}
```

**把上面那段判断也封装进markvalid：**

```c
void Bootloader_MarkValid(void)
{
    boot_info_t bt;
    BootInfo_Load(&bt);
    if(bt.state==OTA_STATE_TESTING)
    {
        //修改bootinfo字段
        bt.state=OTA_STATE_NORMAL;
        bt.boot_count=0;
    }
    BootInfo_Save(&bt);
}
```

## ✅目前IAP项目功能

### boot_count字段：

在`OTA FSM`里面在`OTA_STATE_PENDING`的case分支下对固件进行`CRC32校验`，如果通过CRC32的校验，**写入bootinfo的active_app字段**，**修改state字段为TESTING状态**（这个状态维护boot_count的业务逻辑，在crc成功的时候将boot_count=1，表明是新固件第一次运行）。

设置的次数是前两次上电运行可以正常运行，如果在第三次上电运行前没有**markvalid报平安**,(一直处于**TESTING状态**)，bootloader就会**自动触发回滚**，运行原有的固件。

bootloader模块的OTA状态机里面对bootinfo里面的信息进行处理，**但是不写入**，写入**在跳出switch(state)**以后再进行`boot_save`的操作。

## Flash操作

**擦除/写入flash接口封装：**（32位操作）

```c
void flash_write(uint32_t addr,uint32_t*data,uint32_t len)
{
	HAL_FLASH_Unlock();
	
	for(uint32_t i=0;i<len;i++)
	{
		HAL_FLASH_Program(FLASH_TYPEPROGRAM_WORD,addr+i*4,data[i]);
	}
	
	HAL_FLASH_Lock();
		
}

//擦除页
void flash_erase(uint32_t page_addr)
{
	FLASH_EraseInitTypeDef erase;
	uint32_t page_error;
	
	erase.TypeErase=FLASH_TYPEERASE_PAGES;
	erase.PageAddress=page_addr;
	erase.NbPages=1;
	
	HAL_FLASH_Unlock();
	HAL_FLASHEx_Erase(&erase,&page_error);
	HAL_FLASH_Lock();
	
	
}


```
